## 1. 背景：sgs 的作用

`sgs`（semantic-guess-solver）是一个 Python 库，用于自动求解
**xiaoce.fun GuessWord 每日挑战**。挑战会给出一个目标中文词（通常 2 字），
玩家可以提交猜测；每次猜测返回一个 [0, 1] 区间内的相似度分数
（`doubleScore` 字段），以及一个在猜中目标词时翻转为 `true` 的 `correct` 布尔值。

求解器的任务是 **以尽可能少的猜测次数定位目标词**，
给定一份候选语料（带预计算 BGE-zh-base 嵌入的中文词列表）和一份观测历史
（先前的猜测及其分数）。

### v0.6.0（在多数案例上有效的版本）

v0.6.0 仅内置 **一种排序策略**：分数加权中心点启发式。给定观测 `(word, score)`，计算

```text
centroid = Σ score_i · embedding(word_i) / Σ score_i
```

然后按 `cos(embedding(w), centroid)` 对语料中所有词排序。

它在约 80% 的案例上有效，因为 BGE-zh-base（本语料的嵌入模型）与
预言机的打分模型 *大体* 对齐——在 BGE 空间中语义相近的词，在
xiaoce 空间中也倾向于打出相近的分数。

### v0.7.0（本版本）

剩下约 20% 的案例——尤其是 case-11——会遇到
**嵌入空间错位平台期**，此时中心点会卡在突破阈值（0.95+）之下的某个峰上。
v0.7.0 引入了一个 **核岭回归（KRR）预测器**，直接从观测中学习
BGE → xiaoce 的映射关系，打破该平台期。

## 2. 嵌入空间错位平台期

### 实际表现

```text
Round 1:  peak 0.327 (医院)
Round 2:  peak 0.349 (汉语)         ← 语料扩充略有帮助
Round 3:  peak 0.354 (汉语)         ← 语料扩充开始进入平台期
Round 4:  peak 0.349 (汉语)         ← 仍处于平台期
Round 5:  peak 0.570 (同仁)         ← pivot 至同伴簇，跳升明显
Round 6:  peak 0.654 (同类)         ← pivot 至相似性，继续推进
Round 7:  peak 0.707 (一致)         ← pivot 至一致性
Round 8:  peak 0.753 (允许)         ← pivot 至许可
Round 9:  peak 0.824 (通过)         ← pivot 至放行
Round 10: peak 0.824 (通过)         ← 停滞
Round 11: peak 0.824 (通过)         ← 停滞
Round 12: peak 0.824 (通过)         ← 停滞
Round 13: peak 0.824 (通过)         ← 停滞
Round 14: peak 0.989 (继续)         ← KRR 突破！
```

注意规律：**前 9 轮呈稳定上升**（每次 pivot 大约带来 0.1 的峰分提升）。
Round 9 之后，峰分在额外 5 轮探测中始终卡在 0.824。

这就是 **嵌入空间错位平台期**：
- 中心点尽职工作：识别出答案位于抽象动词簇中（通过/履行/允许/同意/执行）。
- BGE-zh-base 把这个簇**与一片更大的语义区域混在一起**，那片区域还包含
  继续/停止/结束/决定——但中心点作为一个加权均值，只能指向 BGE 空间中
  一个方向。
- 0.824 的天花板，就是 **单个向量** 在 xiaoce 打分空间上所能解释的上限。

### 根本原因

xiaoce.fun **并未公开其嵌入模型**。根据经验观察，它很可能是一个
针对词相似度任务训练的中文 BERT 变体，例如以下之一：
- `shibing624/text2vec-base-chinese`
- `BAAI/bge-large-zh-v1.5`（与我们用的 `bge-base-zh-v1.5` 不同）
- 某个自定义微调模型

无论是哪一个，**BGE-zh-base 与 xiaoce 模型对中文词的摆放方式都不同**。
具体来说，在 BGE-zh-base 空间中，下面这些词

```text
通过 (pass), 履行 (fulfill), 允许 (allow), 同意 (agree),
执行 (execute), 答允 (consent), 拒绝 (refuse)
```

构成 **一个紧密的簇**（两两余弦约 0.85）。而在 xiaoce 空间中，它们
构成 **两个分开的簇**：

```text
{通过, 履行, 允许, 同意, 执行}     ← xiaoce 簇 A（高分：0.65-0.82）
{答允, 拒绝, 同意}                  ← xiaoce 簇 B（较低分：0.40-0.55）
```

中心点在 BGE 空间中找到的是簇 A 的中心，但那个中心**同时**也是
BGE 空间中 `{答允, 拒绝}` 的中心（因为 BGE 把它们混在了一起）。
于是中心点排序语料时，会把簇 B 的词也排得与簇 A 一样高——但
xiaoce 只会给它们 0.4 分，而非 0.8 分。

**平台期就是这种错位的指纹。**

## 3. 为何中心点会停滞——一个具体例子

来看 case-11 在平台期附近的实际观测：

| Word | xiaoce score |
|---|---|
| 通过 (pass) | 0.8245 |
| 履行 (fulfill) | 0.6790 |
| 允许 (allow) | 0.7534 |
| 同意 (agree) | 0.5661 |
| 答允 (consent) | 0.2805 |
| 服从 (obey) | 0.5982 |
| 一致 (consistent) | 0.7072 |
| 相同 (same) | 0.6502 |

这 8 条观测的中心点（BGE 空间）指向这 8 个嵌入在 BGE 空间中的 **均值**。
由于 `通过` 在加权和中的权重最高（0.8245），该均值 **更靠近** `通过`
而不是任何具体的簇内同伴。

但任意候选词的 **真实** xiaoce 分数取决于它属于 *哪个子簇*（上文的 A 或 B）。
中心点无从知晓子簇的存在，因为 BGE 把它们混在了一起。

**结果**：中心点排得高的候选词，与中心点的相似度都在 [0.78, 0.85] 区间，
而这些候选词的 xiaoce 分数横跨 [0.4, 0.85]——在排名顶端，中心点相似度
与 xiaoce 分数之间并不存在单调关系。

这就是在同一抽象动词簇内扩语料没有用的原因：每个新候选词都处于同一 BGE
区域，中心点相似度相近，xiaoce 分数也相近（在先前峰分 ±0.05 范围内）。

## 4. 核岭回归前来救场

### 思路

如果中心点无法分离子簇，我们就需要一个能 **从数据中学习子簇边界** 的模型。
BGE 嵌入作为特征，xiaoce 分数作为目标。在这些 `(X, y)` 对上拟合一个
有监督回归器，应当能恢复 BGE → xiaoce 的映射。

### 为何偏偏是 KRR

核岭回归（Kernel Ridge Regression, KRR）就是 **核诱导特征空间中的岭回归**。
配上 RBF 核时，它等价于一个不带贝叶斯开销的高斯过程回归。
闭式解为 `(K + α·I)^(-1) y`，其中 `K` 是核矩阵，`α` 是岭惩罚。

KRR 具备我们想要的三个性质：
1. **非线性**：即便 BGE → xiaoce 映射不是线性投影，也能学得到。
2. **闭式**：无需迭代训练、无需早停、无需学习率。一次矩阵求解即可。
3. **开箱即用**：`γ=0.1, α=0.1` 在不同嵌入维度与样本量下都表现良好，无需调参。

### 为何不是线性回归

BGE 768 维 → xiaoce 分数上的线性回归，在 case-11 上 5 折交叉验证 MAE 约 0.18。
RBF KRR 的 MAE 约 0.099。之所以非线性很重要，是因为 BGE → xiaoce 映射
至少存在一个"方向翻转"——BGE 拉得近的词，其 xiaoce 分数并不随余弦单调变化。

### 为何不是神经网络

一个 768→128→1 的 MLP 能工作，但需要：
- 训练循环（epoch、batch、早停）
- 用于选超参数的验证集
- GPU 才能在合理时间内训练
- 重新实现嵌入加载器、优化器、损失函数

对我们的规模而言，KRR **更简单、更快**：700 条观测、2257 个候选、
一次矩阵求解。CPU 上总耗时约 30ms。

## 5. KRR 实现

`sgs/krr.py` 导出三个公开符号：

```python
def rbf_kernel(X1: np.ndarray, X2: np.ndarray, *, gamma: float) -> np.ndarray:
    """K[i,j] = exp(-gamma * ||X1[i] - X2[j]||^2)."""

class KernelRidgePredictor:
    """Fitted KRR model. Callable: pred(X) → scores."""

def fit_predictor(
    X_obs: np.ndarray,
    y_obs: np.ndarray,
    *,
    gamma: float = 0.1,
    alpha: float = 0.1,
) -> KernelRidgePredictor:
    """Fit a KRR model from observations."""
```

### 数学（速览）

给定观测 `(x_1, y_1), ..., (x_n, y_n)` 与 RBF 核
`K(x, x') = exp(-γ ||x - x'||^2)`，KRR 拟合函数

```text
f(x) = Σ_i α_i K(x, x_i)
```

其对偶系数为

```text
α = (K + α·I)^(-1) y
```

其中 `K_{ij} = K(x_i, x_j)` 是 `(n, n)` 核矩阵，`y = (y_1, ..., y_n)` 是
目标向量。新点 `x` 处的预测就是 `f(x)`。

### 数值实现选择

| 选型 | 原因 |
|---|---|
| 核矩阵与求解过程使用 `float64` | 当 X 为 768 维时，`float32` 在平方距离步骤会下溢 |
| 使用 `np.linalg.solve` 而非 `cho_solve` | `cho_solve` 在 `scipy.linalg` 中；sgs 仅依赖 numpy |
| 对核矩阵做对称化 `0.5 * (K + K.T)` | 叉积步骤中的数值抵消可能让 `K` 出现轻微不对称 |
| 拟合前 `np.clip(y, 0, 1)` | 防御异常的回放记录 |
| 预测后 `np.clip(predictions, 0, 1)` | RBF 可能外推到 [0, 1] 之外 |
| 求解失败时回退到 `np.linalg.lstsq` | 罕见——仅在极端 (γ, α) 下 `K` 退化时发生 |

### 数值稳定性（测试验证）

对 `n = 1000` 条观测、`γ = 0.1, α = 1.0`，核矩阵条件良好
（`condition_number < 10^4`），且预测值有限（`np.isfinite(out).all()`）。
见 `tests/test_krr.py::test_kernel_matrix_no_overflow_at_n1000`。

## 6. 超参数：为何选取 (γ=0.1, α=0.1)

### 网格搜索

在 case-11 的 701 条观测上做 5 折交叉验证：

| γ \ α | 0.01 | 0.1 | 1.0 | 10.0 |
|---|---|---|---|---|
| 0.01 | 0.0996 | 0.1115 | 0.1479 | 0.1656 |
| **0.1** | 0.1106 | **0.0992** | 0.1125 | 0.1492 |
| 1.0 | 0.1015 | 0.0994 | 0.1012 | 0.1271 |
| 10.0 | 0.2822 | 0.2838 | 0.2925 | 0.3021 |

5×5 网格在 `(γ, α) ∈ {0.01, 0.1, 1.0}^2` 上基本平坦（MAE 差异在 ±10% 内）。
`γ = 10` 过于局部（核函数只在极小邻域内衰减，忽略大部分观测）。
`α = 10` 正则过强（对偶系数被稀释）。

优胜组合 `(0.1, 0.1)` MAE 最低，且在 (γ, α) 轻微扰动下 **稳定**。
亚军 `(1.0, 0.1)` 几乎打平；我们选 `(0.1, 0.1)` 是因为其核衰减更平缓，
在候选词 *远离* 所有观测时（case-11 的典型场景）泛化更好。

### 为何这套默认值应该具有泛化性

BGE-zh-base 嵌入分布下，不相关词之间的典型余弦相似度约 0.5，
相关词之间约 0.8。取 `γ = 0.1`，两个单位范数向量在余弦为 `c` 时的核值为

```text
d^2 = 2 - 2c
K = exp(-0.2 + 0.2c) = e^(-0.2) · e^(0.2c) ≈ 0.82 · e^(0.2c)
```

所以 `c = 0.8`（相关）时 `K ≈ 0.98`，`c = 0.5`（不相关）时 `K ≈ 0.91`。
动态范围窄但 **可预测**：相关与不相关之间约 7% 的差距。配上足够强的岭正则，
足以驱动排序决策。

对于不同的嵌入模型（例如余弦分布更紧凑的 `text2vec-base-chinese`），
`γ` 可能要取 `0.5` 或 `1.0`。默认值针对 BGE-zh-base 768 维调优，
这也是 sgs 的首选嵌入。

## 7. 集成到 sgs.rank 与 round1 CLI

### `sgs.rank.rank_by_predictor`

作为 `sgs.rank.rank` 的直接替代：

```python
from sgs.rank import load_corpus, rank, rank_by_predictor
words, emb = load_corpus("data/cand_words.json", "data/cand_emb.npy")
obs = [("通过", 0.8245), ("履行", 0.6790), ("允许", 0.7534), ...]

# 中心点排序（v0.6.0 行为）
top_centroid = rank(obs, words, emb, top_k=30)

# KRR 排序（v0.7.0）
top_krr = rank_by_predictor(obs, words, emb, top_k=30, gamma=0.1, alpha=0.1)
```

输入相同，排序函数不同。`rank_by_predictor` 按需懒加载 `sgs.krr`，
因此只使用中心点启发式的调用方不会引入 KRR 相关的代码。

### `python -m sgs.round1 --predictor`

CLI 新增 `--predictor` 标志。默认行为（中心点）保持不变以向后兼容。

```bash
# 中心点排序（v0.6.0 默认）
python -m sgs.round1 \
  --replay replay/377580436223.ndjson \
  --candidates data/cand_words.json \
  --embeddings data/cand_emb.npy \
  --batch-size 30

# KRR 排序（v0.7.0）
python -m sgs.round1 \
  --replay replay/377580436223.ndjson \
  --candidates data/cand_words.json \
  --embeddings data/cand_emb.npy \
  --batch-size 30 \
  --predictor

# KRR + 自定义超参数
python -m sgs.round1 \
  --replay replay/377580436223.ndjson \
  --candidates data/cand_words.json \
  --embeddings data/cand_emb.npy \
  --predictor --gamma 0.5 --alpha 0.05 \
  --batch-size 30
```

### 向后兼容

`rank()` 未做任何变更。所有现有调用方（测试、脚本、`solve` CLI 驱动）
继续工作。新代码路径通过 `--predictor` 标志或 `rank_by_predictor` 函数显式开启。

## 8. 何时从中心点切换到 KRR

### 触发规则（3 个条件，全部满足）

1. **中心点峰分 > 0.5 持续 ≥ 3 轮。** 求解器已识别出非随机簇。KRR 需要
   一致的信号才能拟合；否则只是在记忆噪声。
2. **中心点峰分 < 0.85 持续 ≥ 3 轮。** 平台期是真实的，而不是单轮偏差。
   峰分 0.89 意味着中心点 *几乎* 突破——暂时不切换，再多探几条。
3. **观测总数 > 100。** KRR 需要足够样本来拟合一个非平凡的核函数。
   低于 100 时，5 折交叉验证 MAE 不可靠。

### case-11 触发时间线

| Round | Peak | Obs count | 触发？ |
|---|---|---|---|
| 1 | 0.327 | 53 | ✗（peak < 0.5） |
| 2 | 0.349 | 73 | ✗（peak < 0.5） |
| 3 | 0.349 | 103 | ✗（peak < 0.5） |
| 4 | 0.349 | 143 | ✗（peak < 0.5） |
| 5 | 0.570 | 173 | ✗（仅 1 轮 > 0.5） |
| 6 | 0.654 | 274 | ✓（peak > 0.5 已持续 2 轮） |
| 7 | 0.707 | 378 | ✓（3 轮 > 0.5） |
| 8 | 0.753 | 476 | ✓ 此处本应触发——KRR 大约 30 次探针即可求解 |

按 case-11 的轨迹，触发本应在 Round 7 结束时点火（peak 0.707，378 条观测，
平台期已稳定 3 轮）。而实际求解发生在 7 轮之后、254 次额外探针——
若采用触发规则，可节省约 250 次探针与约 4 分钟。

### 反触发：何时 *不要* 切换

- **中心点峰分 > 0.85 且方差高**：平台期较浅，继续探测很可能有效。
- **观测总数 < 30**：冷启动场景，没有信号供 KRR 拟合。使用中心点 + 语料扩充。
- **全零平台期（peak < 0.1）**：完全没有信号，任何模型都帮不上。请用户给提示。

## 9. 局限性与未来工作

### 已知局限

1. **每次会话重新拟合**：`rank_by_predictor` 每次调用都从零开始重训 KRR。
   对于 100+ 轮的主动学习循环，总代价为 `O(rounds · n_obs^3)`（矩阵求解）。
   经验上对 `n_obs ≤ 1000` 没问题（每次约 30ms），但到 `n_obs = 10000`
   时单次求解变为 30 秒。未来的优化方向：维护一个 `KernelRidgePredictor`
   实例，调用 `update()` 而不是重新拟合。
2. **单遍 KRR**：未使用贝叶斯模型平均或不确定性量化。配 RBF 的 KRR 本质上
   是点估计拟合。若想要更丰富的不确定性信号，高斯过程回归是自然的升级。
3. **一次只处理一个平台期**：`rank_by_predictor` 在所有观测上拟合单一模型。
   若观测历史中包含 *两个* 不同的簇（例如会话中途 pivot），KRR 会取平均。
   未来增强方向：检测多簇观测，分别拟合 KRR 模型。

### 未来工作

- **在线 KRR**：新观测到来时增量更新，避免完整矩阵重训。
- **多核 KRR**：学习 RBF / 线性 / 多项式核的凸组合，由 5 折交叉验证挑选。
- **按簇 KRR**：在观测历史中检测簇边界，分别拟合模型。
- **GPU 加速**：对 `n_obs > 5000`，将核矩阵计算批量化到 GPU。

## 附录 B：KRR 对偶形式的数学推导

### 设置

给定训练数据 `(x_1, y_1), ..., (x_n, y_n)`，其中 `x_i ∈ R^d`，
`y_i ∈ R`。我们要拟合形如

```text
f(x) = Σ_i α_i K(x, x_i)
```

的函数 `f: R^d → R`，其中 `K` 是正定核（在我们的场景下为 RBF）。
`α_i` 即我们要求解的对偶系数。

### 特征空间中的岭回归

定义特征映射 `φ: R^d → H`，其中 `H` 是 Hilbert 空间，
满足 `K(x, x') = <φ(x), φ(x')>`。在 `H` 中，函数是线性的：

```text
f(x) = <w, φ(x)>,   w = Σ_i α_i φ(x_i)
```

在 `H` 上的岭回归最小化

```text
L(w) = (1/n) Σ_i (f(x_i) - y_i)^2 + (λ/2) ||w||^2
       = (1/n) ||Φα - y||^2 + (λ/2) α^T K α
```

其中 `Φ = [φ(x_1), ..., φ(x_n)]`，`K = Φ^T Φ` 是核矩阵。
令梯度为零：

```text
(1/n) Φ^T (Φα - y) + λ K α = 0
(1/n) (K α - K y) + λ K α = 0     (由 Φ^T Φ = K 与 Φ^T y = K y)
(1/n) K y = (1/n) K α + λ K α = K α ((1/n) + λ)
α = ((1/n) + λ)^(-1) (1/n) y
```

等等，这是简化形式。让我们更仔细地重做一遍——标准形式使用不同的参数化。

### KRR 闭式解

定义 **原始权重** `w` 与 **对偶系数** `α`，使得 `w = Σ_i α_i φ(x_i)`。
表示定理（representer theorem）保证此形式对核岭是最优的。

矩阵形式下，KRR 问题为

```text
min_α  (1/2) α^T K^2 α - α^T K y + (1/2) y^T y + (λ/2) α^T K α
```

对 α 求导并令其为零：

```text
K^2 α - K y + λ K α = 0
K α (1 + λ K^(-1)) ... wait, that's not quite right.
```

我们以规范形式重新开始。标准 KRR 推导：

```text
α* = argmin_α  (1/2) ||K α - y||^2 + (λ/2) α^T K α
```

令梯度为零：

```text
K (K α - y) + λ K α = 0
(K + λ I) α = y
α = (K + λ I)^(-1) y
```

这正是 `sgs.krr.fit_predictor` 中使用的形式：

```python
A = K.astype(np.float64) + alpha * np.eye(n, dtype=np.float64)
dual_coef = np.linalg.solve(A, y)
```

代码中的 `alpha` 参数对应标准形式的 `λ`——岭正则强度。

### 预测

对于新点 `x`：

```text
f(x) = Σ_i α_i K(x, x_i) = k(x)^T α
```

其中 `k(x) = [K(x, x_1), K(x, x_2), ..., K(x, x_n)]^T`。

对一批 `X_test` 写成矩阵形式：

```text
f(X_test) = K_test α
```

其中 `K_test[i, j] = K(X_test[i], X_train[j])`。

这正是 `sgs.krr.KernelRidgePredictor.__call__` 所计算的内容。

### 为何 KRR 有效而中心点不行

中心点使用观测的 **单一加权均值**：`c = Σ score_i · x_i / Σ score_i`。
对新点的预测是 `cos(x, c) = <x, c>`。

KRR 使用观测的 **RBF 核加权和**：`f(x) = Σ α_i exp(-γ ||x - x_i||^2)`。
每条观测以各自学到的权重 `α_i` 贡献，贡献随距离平滑衰减。

关键差异：即便两条观测的 xiaoce 分数相同，KRR 也能 **对一条赋正权重、
对另一条赋负权重**。这意味着 KRR 能 *减去* 那些让中心点产生混淆的
BGE 簇结构，从而把 xiaoce 的专属信号剥离出来。

在 case-11 中，对 `{答允, 拒绝, 答允}`（xiaoce 分数 0.28–0.55）赋的负权重，
抵消了与 `通过, 履行, 允许` 的 BGE 空间对齐，留下一个指向
`继续, 停止, 结束, 决定` 的残差——这些词中心点永远无法排到高位，
因为它们在 BGE 空间中 *远离* 中心点。

这就是 KRR 打破平台期的 **代数本质**。
